

# 全天录音智能摘要系统技术方案（Qwen3-Max版）

## 1. 系统架构设计

### 1.1 整体流程概览

#### 1.1.1 数据输入层：ASR文本与说话人分离结果接入

数据输入层承担接收DingTalk A1或钉钉AI听记等设备输出的核心职责，需处理两类异构数据的融合：**ASR自动语音识别结果**（时间戳、识别文本、置信度）与**Speaker Diarization说话人分离结果**（说话人标识、声纹聚类、起止时间）。系统采用统一数据模型，将两者对齐为"时间-说话人-内容"三元组序列，以**500毫秒**为粒度进行时间切片，合并连续窗口生成完整的说话片段（utterance）。选择500ms而非更细粒度，是因为ASR输出本身的时间戳精度有限，过细的对齐粒度（如200ms）不会带来信息增益，反而增加碎片化风险。

全天无暂停录制的数据规模估算：假设8-12小时录音，考虑**40-60%有效说话比例**（扣除静默、重叠、环境噪声），实际有效文本量约**3万-8万汉字**，折合**4.5万-12万tokens**。Qwen3-Max上下文窗口为**262,144 tokens**，最大输出**65,536 tokens**（启用思考模式时为32,768 tokens），这意味着约**90%的工作日**可在单次调用中处理全量文本，无需分片。这一模型能力从根本上简化了系统架构。

数据模型在核心三元组之外，增加**不确定性标注字段**以支撑下游决策：

| 字段 | 类型 | 含义 | 取值范围 |
|:---|:---|:---|:---|
| `speaker_confidence` | float | 说话人归属置信度 | 0.0-1.0 |
| `asr_confidence` | float | ASR识别置信度 | 0.0-1.0 |
| `alignment_quality` | enum | ASR与说话人分离的时间对齐质量 | `high` / `medium` / `low` |

输入层还需处理设备格式差异。DingTalk A1输出JSON格式，包含`device_id`、`session_start`、`session_end`、`segments`数组；钉钉AI听记额外提供`speaker_name`推测与`is_overlap`标记。系统通过**适配器模式**封装差异，对外暴露统一接口，确保核心逻辑与设备类型解耦。

#### 1.1.2 预处理层：结构化解析与特征提取

预处理层将原始数据转换为富含特征的结构化表示，支撑后续智能分层决策。处理流程包含五个环节：

| 环节 | 功能 | 关键技术 |
|:---|:---|:---|
| 对话会话检测 | **第零层切分**：按静默>3min且说话人变化识别自然会话边界 | 双条件门控（静默时长 + 说话人切换） |
| 数据清洗 | 过滤低置信度片段、修正时间戳异常、标准化标点数字 | **分级置信度阈值**（见下文），正则规则库 |
| 话语合并 | 聚合短碎片为语义连贯单元 | 3-5秒间隔阈值，说话人连续性判断 |
| 特征计算 | 提取时间、说话人、结构、文本统计特征 | 时长分布、轮次序号、关键词密度 |
| 元数据关联 | 建立说话人到关键人配置的匹配链接 | 多模式匹配（精确/纠错/模糊/内容推断） |

**置信度阈值分级策略**：统一阈值0.6会误删关键人的低质量识别片段（嘈杂环境中关键指令往往识别置信度偏低）。系统采用分级方案：

| 说话人级别 | `asr_confidence`阈值 | 设计理由 |
|:---|:---|:---|
| P0/P1关键人 | **0.35-0.4** | 宁可保留噪声，不可丢失关键信息 |
| P2关键人 | **0.50** | 适度平衡 |
| 非关键人 | **0.6** | 过滤低质量片段，降低下游处理负担 |

合并后的对话单元时长中位数约**20-60秒**，设置**120秒合并上限**——超过此阈值的连续语段强制拆分，避免产生过长的不可分割单元影响后续分片灵活性。关键元数据包括：时段标记（工作/休息）、场景推测（基于关键词）、以及关键人匹配结果（精确匹配、别名匹配、职务推断）。

#### 1.1.3 智能分层：关键人识别与片段重要性评分

智能分层模块是系统的**核心决策引擎**，实现"关键人优先+语义评估"的筛选策略。关键人识别采用**四级匹配策略**（详见§3.2）：

| 级别 | 匹配方式 | 适用场景 | 准确率 |
|:---|:---|:---|:---|
| L1 | Speaker ID精确匹配 | 已建立声纹映射的稳定场景 | >95% |
| L1.5 | ASR人名纠错匹配 | 同音字/形近字修正 | 90-95% |
| L2 | 姓名/别名模糊匹配 | 设备提供speaker_name或文本自报 | 85-90% |
| L3 | 内容推断匹配（标记为疑似） | 首次录制或识别失败时的降级保障 | 70-85% |

片段重要性评分采用**统一的MVP公式**（详见§4.1）：

$$\text{FinalScore} = \text{KeyPersonBaseScore} + \text{LLMScore} \times 10$$

其中**KeyPersonBaseScore**基于等级映射（P0=80, P1=50, P2=20, P3=0）。P0级关键人基础分80天然确保进入候选池（阈值60），实现**短对话保护**——"总裁5分钟指示"不会因时长过短被过滤。

#### 1.1.4 摘要生成层：两级弹性架构与模型调用

摘要生成层基于Qwen3-Max的262K上下文能力，采用**两级弹性架构**，根据当日数据量自适应选择处理路径：

| 路径 | 触发条件 | 处理方式 | 适用比例 |
|:---|:---|:---|:---|
| **短日直通** | 有效内容 <80K tokens | 全量文本单次调用，直接生成完整日报 | **~90%工作日** |
| **长日分片** | 有效内容 80K-250K tokens | 按时段分2-3个片段 → 片段摘要 → 合并生成日报 | ~10%工作日 |
| **降级三级** | 模型不可用或回退至小模型 | 片段级→时段级→全天级三级渐进压缩 | 异常降级路径 |

**短日直通路径**（核心路径）：

将全部预处理后的文本、关键人配置、Prompt指令打包为单次请求。Qwen3-Max在262K上下文中可充分理解全天对话的完整脉络，避免分片导致的信息割裂。输出采用思考模式（最大输出32,768 tokens），一次性生成结构化日报。

**长日分片路径**：

当数据量超过80K tokens（约占10%的高密度工作日），按自然时段（上午/下午/晚间或按对话会话）分为2-3个片段。每个片段独立生成摘要后，通过一次合并调用生成最终日报。片段之间采用**尾部上下文摘要**字段——每个片段的输出携带一段200-300字的"上下文接力"摘要，供下一片段或合并阶段参考，既保证连贯性又避免重复输入浪费token。

**降级三级路径**（容灾备用）：

保留原三级渐进压缩架构（片段级→时段级→全天级），用于Qwen3-Max不可用时回退至小上下文模型的场景。每级压缩比5-10倍，总体压缩比达100:1以上。

**Token预算分配**（以262,144 tokens上下文为基准）：

| 项目 | 短日直通 | 长日分片（单片） | 合并调用 |
|:---|:---|:---|:---|
| 系统提示词 | 1,000-1,500 | 800-1,000 | 1,000-1,200 |
| 关键人配置注入 | 400-600 | 300-400 | 400-500 |
| 有效内容载荷 | **≤200,000** | 60,000-80,000 | 15,000-25,000 |
| 尾部上下文字段 | — | 500-800 | 1,000-2,000 |
| 输出预留 | 30,000-32,768 | 8,000-16,000 | 30,000-32,768 |
| 安全余量 | ≥20,000 | ≥5,000 | ≥5,000 |

#### 1.1.5 输出层：结构化日报与关键信息归档

输出层生成**分层信息架构**的日报产物（详见§5.4），分为核心层与扩展层：

- **核心层（必生成）**：执行摘要（50-100字）+ 关键会议回顾（含决策）+ 行动事项总表
- **扩展层（可选）**：悬置议题 + 跟进建议 + 明日已知安排

系统提供三种阅读粒度：**30秒速览版**（~100字）、**3分钟标准版**（500-800字）、**完整存档版**（1,500-3,000字）。

归档系统建立**多维度索引**（时间、人物、主题、项目），支持90天内的全文检索。原始ASR、各级摘要、评分记录、用户反馈均持久化存储，形成**数据飞轮**驱动持续优化。

### 1.2 核心组件模块

#### 1.2.1 配置管理模块：关键人列表加载与更新

配置管理采用**轻量化方案**，关键人配置 `key_people.yaml` 纳入 Git 仓库统一管理：

```yaml
# config/key_people.yaml
version: "2026-03-27"
people:
  - id: "kp001"
    name: "张三"
    aliases: ["张总", "总裁", "CEO"]
    level: "P0"
    speaker_ids: []
    short_dialog_protection: true
    always_include: true
  - id: "kp002"
    name: "李四"
    aliases: ["李总监"]
    level: "P1"
    speaker_ids: ["spk_042"]
    short_dialog_protection: false
    always_include: false
```

**加载策略**：启动时读取并缓存 → 手动 `POST /admin/reload-config` 或 CLI `reload` 触发重新加载 → 校验失败保留旧配置。变更通过 Git commit 追踪，回滚通过 `git revert` + reload 实现（详见§3.1）。

#### 1.2.2 分片策略引擎：会话检测、多信号边界与关键人保护

分片策略引擎采用**分层插件化架构**，引入"对话会话检测"作为第零层，并支持多种策略的动态组合：

**第零层：对话会话检测**

在所有分片策略之前，系统首先识别自然对话边界。检测条件为**静默>3分钟**且伴随**说话人变化**（双条件门控，避免单人长时间思考被误切）。会话检测的输出作为后续所有分片策略的硬约束——任何分片不会跨越会话边界。

**碎片短对话聚合**：对于<5分钟的非关键人短对话片段，系统自动将其**打包聚合**为一个处理单元。聚合逻辑按时间邻近性分组，单个聚合包不超过15分钟，避免碎片化对话浪费独立的LLM调用。

| 策略层 | 核心机制 | 适用场景 | 优先级 |
|:---|:---|:---|:---|
| L0 会话检测 | 静默>3min + 说话人变化 | **所有场景的预切分** | **硬约束** |
| L1 关键人保护 | 关键人对话保护区，内部再细分 | P0/P1级关键人出现 | **优先约束** |
| L2 主题边界 | **多信号融合检测**（见下文） | 议题清晰的会议场景 | 中等 |
| L3 时间窗口 | 固定30-60分钟切片，软边界调整 | 默认保底策略 | 最低 |

**关键人保护区规则**：关键人保护区作为"优先约束"而非"硬性取代"——保护区边界在与主题边界、时间窗口冲突时**优先保留**，但当主题边界与保护区边界高度吻合（偏差<90秒）时，采用主题边界以获得更自然的切分点。超长保护区（>60分钟）内部采用主题边界或时间窗口再细分，同时标注关联性（如"战略研讨会-第2部分，共4部分"）。

**主题边界检测——多信号融合**：

单一信号（如纯语义相似度）在口语场景中噪声较大。系统采用多信号融合方案：

| 信号 | 权重 | 检测方式 |
|:---|:---|:---|
| 说话人切换 | 0.35 | 连续说话人集合发生显著变化（Jaccard距离>0.5） |
| 静默间隔 | 0.35 | 当前间隔显著超过局部均值（>2倍标准差） |
| 语义辅助 | 0.30 | 相邻窗口的文本embedding余弦相似度低于阈值 |

三信号加权分数超过阈值时判定为主题边界。该方案避免了对单一语义模型的过度依赖，在ASR文本质量参差的场景下更为鲁棒。

**软边界搜索范围**：分片切分点在初始候选位置的**±60-90秒**范围内搜索最佳边界（自然句末、静默间隔、说话人轮换）。

#### 1.2.3 Token预算控制器：弹性适配Qwen3-Max 262K上下文

Token预算控制器从"严格合规固定上限"转变为**弹性适配**策略，核心机制：

- **精确分词**：集成Qwen3官方tokenizer，实时计算输入token数
- **路径选择**：有效内容载荷 ≤200,000 tokens时走短日直通路径；超过时触发长日分片
- **分片触发阈值**：**200,000 tokens**（预留62K用于Prompt、输出、安全余量）
- **内容压缩**：分片不可行时，按重要性降序截断，保留关键内容优先
- **超限防护**：三级防护（路径选择→分片→压缩→人工队列）

成本监控实时追踪消耗，按**摘要级别、时段、关键人**维度聚合，设置日/月预算告警阈值。由于短日直通路径减少了多次调用开销，预计整体API成本较三级压缩方案**下降40-60%**。

#### 1.2.4 重要性评估模块：MVP统一评分

重要性评估采用**统一的MVP公式**（详见§4），核心由两个变量驱动：

- **KeyPersonBaseScore**：基于关键人等级的规则映射（P0=80, P1=50, P2=20, P3=0）
- **LLMScore**：轻量级LLM批量调用，1-10分语义评估

在短日直通路径下，重要性评估主要用于**输出层的展示排序**和**执行摘要的重点选取**，而非输入端的内容裁剪。仅在长日分片路径中，评估结果才用于指导分片内的内容优先级排序。

完整阶段将引入DurationScore、ContextScore、TimePeriodCoeff等维度，权重通过用户反馈数据回归确定。

#### 1.2.5 摘要融合模块：层级化摘要合并与去重

摘要融合模块在不同路径下承担不同职责：

- **短日直通路径**：融合模块主要负责**输出格式化**和**结构一致性校验**，无需跨片段合并
- **长日分片路径**：融合模块承担跨片段信息一致性、冗余消除、叙事连贯性三大核心任务

| 功能 | 实现机制 | 输出 |
|:---|:---|:---|
| 跨片段一致性 | 关键事实溯源、数值校验、矛盾检测标记 | 带来源标注的摘要 |
| 语义去重 | text2vec-base-chinese / m3e-large 向量相似度 **>0.90-0.92** 时合并，辅以**实体级校验** | 去重后的摘要集合 |
| 尾部上下文衔接 | 利用片段携带的上下文接力字段恢复跨片段语境 | 连贯的跨片段叙事 |
| 关联建立 | 通过 topic_id 建立主题关联、人员关联、因果关联、时间关联 | 关联图谱或嵌入文本 |
| 叙事优化 | 时间线组织、过渡语句、framing设计 | 连贯可读的最终日报 |

## 2. Qwen3-Max模型约束与适配策略

### 2.1 模型规格参数

#### 2.1.1 上下文窗口：262,144 tokens大容量窗口

Qwen3-Max提供**262,144 tokens上下文窗口**（最大输入258,048 tokens，系统/输出预留4,096 tokens），为会议摘要场景提供了充裕的处理空间。实际可用容量需扣除：

- 系统提示词：800–1,200 tokens
- 关键人配置注入：200–500 tokens
- 输出预留：根据模式不同，4,000–65,536 tokens（详见 2.1.2）

**有效内容载荷可达190,000–250,000 tokens**，利用率极高。中文场景下，按**1.5–1.8字符/token**编码效率估算，单次调用可承载约**285,000–450,000汉字**，覆盖绝大多数全天会议转录文本。

注意力机制特性要求关键信息放置在**输入后部**（靠近提示词），以最大化模型关注度。在超长输入场景中，建议将关键人对话和高优先级内容集中放置于输入末端。

#### 2.1.2 输出限制：双模式差异化上限

Qwen3-Max支持**非思考模式**和**思考模式**两种推理方式，输出上限存在显著差异：

| 模式 | 最大输出tokens | 思维链上限 | 适用场景 |
|:---|:---|:---|:---|
| 非思考模式 | 65,536 tokens | — | 常规摘要生成、结构化提取 |
| 思考模式 | 32,768 tokens（回答） | 81,920 tokens（思维链） | 复杂推理、跨片段关联分析 |

**输出配置建议**：

- 片段级摘要：`max_tokens=4,000`，非思考模式，满足细节提取需求
- 时段级摘要：`max_tokens=6,000`，非思考模式，整合多片段结论
- 全天级摘要：`max_tokens=8,000`，可启用思考模式以增强全局洞察，配合 `thinking_budget=16,000` 控制推理深度
- 后校验检测JSON完整性，触发重试或修复流程

**thinking_budget 控制建议**：思考模式下，通过 `thinking_budget` 参数限制思维链长度，避免不必要的推理开销：

| 任务类型 | thinking_budget | 说明 |
|:---|:---|:---|
| 简单提取/格式化 | 不启用思考模式 | 无需推理链 |
| 跨片段整合 | 8,000–16,000 | 中等推理深度 |
| 全天决策洞察 | 16,000–32,000 | 深度推理，发掘隐含关联 |
| 复杂矛盾检测 | 32,000–81,920 | 最大推理深度 |

#### 2.1.3 成本结构：阶梯计价与Batch API优化

Qwen3-Max采用**输入/输出分离、按量阶梯计价**模型（阿里云百炼官方定价）：

**输入定价**：

| 输入量区间 | 单价（元/百万tokens） |
|:---|:---|
| 0–32K tokens | 2.5 |
| 32K–128K tokens | 4.0 |
| 128K–252K tokens | 7.0 |

**输出定价（非思考模式）**：

| 输出量区间 | 单价（元/百万tokens） |
|:---|:---|
| 0–32K tokens | 10.0 |
| 32K–128K tokens | 16.0 |
| 128K–252K tokens | 28.0 |

**输出定价（思考模式）**：统一**28元/百万tokens**（含思维链+回答内容）。

**典型工作日成本估算**：

| 项目 | 输入tokens | 输出tokens | 输入成本 | 输出成本 | 合计 |
|:---|:---|:---|:---|:---|:---|
| 全天会议摘要（单次调用） | 120,000 | 8,000 | ¥0.48 | ¥0.08 | ¥0.56 |
| 全天会议摘要（层级策略） | 160,000 | 14,000 | ¥0.64 | ¥0.14 | ¥0.78 |
| 极端场景（超长会议，分片） | 240,000 | 20,000 | ¥1.36 | ¥0.20 | ¥1.56 |

**Batch API成本优化**：阿里云百炼Batch API可节省**50%费用**。对于非实时场景（如每日定时汇总），强烈建议使用Batch API：

| 策略 | 单日成本（典型） | 月度成本（22天） | 节省比例 |
|:---|:---|:---|:---|
| 实时API | ¥0.78 | ¥17.16 | — |
| Batch API | ¥0.39 | ¥8.58 | 50% |

月度成本（Batch API）约**¥8.58**，具有极佳成本效益。额外优化策略：缓存复用、动态模型降级（低优先级任务切换至Qwen3-Plus）。

### 2.2 262K窗口下的弹性处理策略

#### 2.2.1 策略总览：大多数场景单次调用，极端场景才分片

262,144 tokens的上下文窗口根本性地改变了长文本处理策略。按中文1.5–1.8字符/token估算，单次调用可处理约**28–45万汉字**，覆盖绝大多数8小时工作日的完整会议转录。

**处理策略决策树**：

| 转录文本规模 | 预估tokens | 处理策略 | 说明 |
|:---|:---|:---|:---|
| ≤180,000 tokens（~27–32万字） | ≤180K | **单次调用** | 直接全文输入，无需分片 |
| 180,000–250,000 tokens | 180K–250K | **单次调用 + 精简提示词** | 压缩系统提示词，最大化内容载荷 |
| >250,000 tokens | >250K | **弹性分片** | 仅在极端场景触发，按下述策略分片 |

> **核心原则**：分片是最后手段，而非默认策略。单次调用可避免跨片段信息丢失、重复内容、上下文割裂等问题，应优先采用。

#### 2.2.2 极端场景分片策略：时间窗口 + 主题边界混合切分

仅当单次调用无法容纳全部内容时，启用分片机制：

**时间窗口分片**（保底策略）：默认窗口**120分钟**（自适应范围90–180分钟）。采用**软边界**机制：以目标时间点为基准，±60秒内搜索最近的自然停顿点（标点、静默标记），调整切分边界。

**主题边界分片**（精细策略）：
1. 文本按句子/100字切分为基本单元
2. text2vec-base-chinese / m3e-large 编码为向量
3. 计算相邻单元余弦相似度，绘制时序曲线
4. 识别局部最小值作为候选边界
5. 后处理：合并过近边界（最小片段≥60,000 tokens）、过滤低置信度边界

两种策略**组合使用**：以180分钟为粗粒度框架，内部应用主题边界细分，边界冲突时优先主题边界。

#### 2.2.3 关键人优先策略：关键人对话独立成段不截断

**核心规则**：关键人（尤其P0级）的完整对话**不被任何分片机制切割**。

| 场景 | 处理方式 |
|:---|:---|
| 关键人对话 ≤200,000 tokens | 直接完整输入模型，与其余内容一并处理 |
| 200,000–250,000 tokens | 压缩非关键人内容，确保关键人对话完整输入 |
| >250,000 tokens | 关键人对话独立成片段优先处理，标注关联性 |

> 在262K窗口下关键人对话几乎不可能触发分片，极大简化了处理流程。

#### 2.2.4 层级摘要架构：弹性适配，按需启用

在262K窗口下，层级摘要架构从"必须"变为"可选"：

| 文本规模 | 推荐架构 | 说明 |
|:---|:---|:---|
| ≤180K tokens | **单层直出** | 全文输入，一次生成全天摘要 |
| 180K–250K tokens | **两层架构**（时段→全天） | 按半天分两次输入，再整合 |
| >250K tokens | **三层架构**（片段→时段→全天） | 完整层级摘要流程 |

三层架构的信息压缩规格（仅极端场景适用）：

| 层级 | 信息压缩比 | 核心任务 | 质量保障 |
|:---|:---|:---|:---|
| 片段级 | ~10:1 | 细节提取、行动项识别、关键人高亮 | 抽样人工评估 |
| 时段级 | ~5:1 | 跨片段整合、去重、主题提炼 | 跨片段一致性校验 |
| 全天级 | ~3:1 | 全局视角、跨时段关联、决策洞察 | 全局洞察准确性验证 |

层级间通过**标准化JSON数据结构**传递，支持下层溯源和上层回调。

### 2.3 Token预算弹性分配原则

262K窗口下，Token预算从"严格合规"转向**弹性适配**：

| 项目 | 预算范围 | 占比 | 说明 |
|:---|:---|:---|:---|
| 系统提示词 | 800–1,200 tokens | <0.5% | 角色定义、任务描述、输出格式、示例 |
| 关键人配置注入 | 200–500 tokens | <0.2% | 动态注入，层级递减 |
| 有效内容载荷 | 190,000–250,000 tokens | 73%–96% | 核心变量，绝大多数场景无需分片 |
| 输出响应预留 | 4,000–65,536 tokens | 2%–25% | 根据摘要级别和是否启用思考模式动态调整 |

**弹性策略要点**：
- 当内容载荷 ≤190K tokens时，系统提示词可保持完整版本（含示例和详细格式说明）
- 当内容载荷在190K–240K tokens时，切换为精简版系统提示词（去除示例，仅保留核心指令）
- 当内容载荷 >240K tokens时，触发分片流程，每片段控制在180K tokens以内以保留充足预算

### 2.4 模型版本管理与稳定性保障

#### 2.4.1 版本锁定策略

生产环境**必须使用快照版本**（如 `qwen3-max-2026-01-23`），禁止直接使用 `qwen3-max` 别名。原因：

- 阿里云可能随时更新 `qwen3-max` 指向的底层模型版本
- 模型更新可能导致输出风格、JSON格式遵从度、推理能力等发生变化
- 快照版本确保行为可复现，便于问题排查和质量回归测试

#### 2.4.2 版本升级流程

| 阶段 | 操作 | 验收标准 |
|:---|:---|:---|
| 评估 | 新版本在测试集上运行完整摘要流程 | 输出质量不低于当前版本 |
| 灰度 | 10%流量切换至新版本，持续3个工作日 | 无质量投诉、成本无异常波动 |
| 全量 | 更新配置文件中的模型版本号 | 记录变更日志，保留旧版本回滚能力 |

配置文件中的模型版本应作为**环境变量或配置项**管理，切换时无需修改代码：

```yaml
model:
  name: qwen3-max-2026-01-23   # 快照版本，禁止使用 qwen3-max
  fallback: qwen3-plus          # 降级模型
  thinking_budget_default: 16000
  batch_api_enabled: true
```

## 3. 关键人机制设计

### 3.1 配置文件规范

#### 3.1.1 文件格式：YAML 配置，Git 版本管理

关键人配置采用单一 YAML 文件 `key_people.yaml`，纳入项目 Git 仓库统一管理。通过 Git 天然获得变更历史、diff 对比、分支协作与回滚能力，无需自建版本控制系统。

**MVP 阶段核心字段**：

| 字段 | 类型 | 必填 | 说明 |
|:---|:---|:---|:---|
| `id` | string | 是 | 全局唯一标识 |
| `name` | string | 是 | 正式姓名 |
| `level` | enum | 是 | P0/P1/P2/P3 |
| `aliases` | string[] | 否 | 别名、昵称、职务称谓 |
| `speaker_ids` | string[] | 否 | 关联 ASR Speaker ID |
| `short_dialog_protection` | boolean | 否 | 短对话保护开关（仅 P0/P1 有效） |
| `always_include` | boolean | 否 | 无条件包含标记 |

> **MVP 精简说明**：`effective_from`/`effective_until`（生效/失效日期）和 `pinyin`（拼音变体）字段在 MVP 阶段不纳入，待业务复杂度提升后按需引入。人员变更通过直接修改配置文件并提交 Git 完成。

#### 3.1.2 优先级等级与业务含义

| 等级 | 定位 | 典型角色 | 处理策略 |
|:---|:---|:---|:---|
| **P0** | 决策层 | 总裁、CEO、董事长 | **所有对话保留，保障分 80 确保进入候选池** |
| **P1** | 管理层 | 部门总监、VP | 常规保留，结合 LLM 语义评估排序 |
| **P2** | 协作层 | 项目经理、核心骨干 | 选择性保留，依赖语义重要性 |
| **P3** | 普通员工 | 一般参与者 | 默认过滤，除非语义评分极高 |

#### 3.1.3 配置加载策略：启动加载 + 手动 Reload

配置管理采用**轻量化方案**，避免热更新带来的复杂性：

| 环节 | 策略 | 说明 |
|:---|:---|:---|
| 启动加载 | 服务启动时读取并解析 `key_people.yaml` | 解析失败则阻止启动，输出明确错误 |
| 手动 Reload | 提供 API/CLI 命令触发重新加载 | `POST /admin/reload-config` 或 `cli reload` |
| 校验机制 | Reload 时执行 Schema 校验 + 冲突检测 | 校验失败保留旧配置，返回错误详情 |
| 并发安全 | 读写锁 + 不可变对象原子替换 | 加载期间不影响正在处理的请求 |
| 日志记录 | 每次 Reload 记录时间、配置摘要、变更 diff | 便于问题排查 |

#### 3.1.4 版本控制：Git 原生管理

配置文件 `key_people.yaml` 与代码同仓库管理，变更流程：

1. **修改配置**：编辑 `key_people.yaml`，填写有意义的 commit message
2. **Code Review**：通过 Pull Request 审核变更（推荐）
3. **部署生效**：合并后在目标环境执行手动 Reload
4. **回滚**：`git revert` 对应 commit 后重新 Reload

### 3.2 匹配识别策略

#### 3.2.1 四层匹配架构

系统采用 L1 → L1.5 → L2 → L3 四级匹配流水线，逐层递进：

| 层级 | 策略 | 匹配依据 | 置信度 | 适用场景 |
|:---|:---|:---|:---|:---|
| **L1** | 精确匹配 | 姓名全称、别名库 | >95% | 配置完善、ASR 识别准确 |
| **L1.5** | ASR 人名纠错 | 同音字/形近字映射表 | 90-95% | ASR 常见误识别修正 |
| **L2** | 模糊匹配 | 昵称、职务称谓正则 | 85-90% | 口语变体、称谓推断 |
| **L3** | 内容推断匹配 | 上下文语义推断说话人身份 | 70-85% | 未直呼姓名但可推断 |

**L1.5 ASR 人名纠错层**：在精确匹配与模糊匹配之间插入纠错环节，维护常见 ASR 误识别映射表：

```yaml
# config/asr_name_corrections.yaml
corrections:
  - target: "张总"
    variants: ["章总", "张宗", "掌总"]
  - target: "李明"
    variants: ["黎明", "李鸣", "里明"]
```

纠错表随使用积累持续补充，同样纳入 Git 管理。

**L3 内容推断匹配 — 疑似标记机制**：当仅通过上下文语义推断出说话人身份时，匹配结果标记为 **"疑似（inferred）"**，并赋予折扣权重：

| 原始等级 | 疑似匹配后的等效处理等级 | 说明 |
|:---|:---|:---|
| P0 疑似 | 按 **P1** 待遇处理 | 保障分降级，不享受 P0 无条件保留 |
| P1 疑似 | 按 **P2** 待遇处理 | 降低一级权重 |
| P2 疑似 | 按 **P3** 待遇处理 | 仅语义极高时保留 |
| P3 疑似 | 维持 P3 | 无变化 |

评分分解记录中需明确标注 `match_type: inferred`，供人工复查。

#### 3.2.2 Speaker ID 关联与冷启动方案

**Speaker ID 映射维护**：自动记录新识别的 Speaker ID 与姓名的关联，人工确认后纳入持久映射并写入配置。

**冷启动方案**（系统部署前 5 天）：

| 阶段 | 策略 | 目的 |
|:---|:---|:---|
| 冷启动期（Day 1-5） | 降低匹配过滤阈值至 0.25 | 宁可多保留，避免遗漏关键人 |
| 冷启动期（Day 1-5） | 未匹配到关键人的片段标记为"待确认" | 提示用户人工标注 |
| 冷启动期（Day 1-5） | 每日生成 Speaker ID 映射建议报告 | 加速建立声纹-人名关联 |
| 稳定期（Day 6+） | 恢复正常阈值，关闭额外提示 | 映射库已基本建立 |

冷启动期间，系统在每日摘要末尾附加"Speaker ID 确认请求"，列出当天出现但未关联到关键人的 Speaker ID 及其代表性发言片段，引导用户确认身份。

#### 3.2.3 置信度阈值分级

不同优先级的关键人采用差异化的匹配置信度阈值，降低高优先级人物的漏识别风险：

| 人员类别 | 置信度阈值 | 说明 |
|:---|:---|:---|
| P0/P1 关键人 | **0.35 - 0.40** | 宁可误识别，不可遗漏 |
| P2 关键人 | **0.50** | 适度平衡 |
| 非关键人（P3） | **0.60** | 减少噪声，避免低置信度片段干扰 |

#### 3.2.4 多人对话处理

多人对话的优先级判定：**取所有参与者中的最高等级**，避免"平均效应"稀释关键人。例如，P0 + P2 + P3 的三人对话，片段标记为 P0 级处理。若最高等级来自 L3 疑似匹配，则按疑似降级后的等效等级处理。

## 4. 重要性评估模型

### 4.1 评分公式设计

系统采用**统一评分公式**，消除多套公式并存的歧义。按实施阶段分为 MVP 与完整阶段，MVP 公式为完整公式的子集，确保平滑升级。

#### 4.1.1 MVP 阶段评分公式

$$\text{FinalScore} = \text{KeyPersonBaseScore} + \text{LLMScore} \times 10$$

| 组成部分 | 取值范围 | 说明 |
|:---|:---|:---|
| KeyPersonBaseScore | 0-80 | 关键人等级基础分，体现组织优先级 |
| LLMScore | 1-10 整数 | 轻量 LLM 语义评估，体现内容价值 |
| **总分范围** | **0-180** | KeyPersonBaseScore(0-80) + LLMScore(1-10) × 10 |

**KeyPersonBaseScore 定义**：

| 等级 | 基础分 | 设计意图 |
|:---|:---|:---|
| **P0** | **80** | 保障进入候选池（默认阈值 60），即使 LLMScore 最低(1)也有总分 90 |
| **P1** | **50** | 需要 LLMScore ≥ 2 才能过阈值，中等语义即可保留 |
| **P2** | **20** | 需要 LLMScore ≥ 4 才能过阈值，依赖内容质量 |
| **P3** | **0** | 完全依赖 LLMScore，需 ≥ 7 才能过阈值 |

> **短对话保护已内含**：P0 基础分 80 天然确保短对话不被过滤，无需额外的短对话补偿机制。P0 片段保证进入候选池后，池内仍按 LLMScore 进行**二次排序**，确保高质量对话优先。

**LLMScore 批量调用**：为降低 API 调用成本和延迟，LLM 语义评估采用**批量拼接调用**，每次将 5-10 个片段拼接为一次请求：

```
请评估以下 N 个对话片段的重要性。考虑维度：决策/指令/承诺/争议/紧急事项/任务分配/信息稀缺性。
对每个片段输出 1-10 分整数。

【片段1】...
【片段2】...
...
【片段N】...

请按顺序输出 N 个分数，用逗号分隔，仅输出数字。
```

调用参数：非思考模式，max_tokens=64，temperature=0。每批 5-10 个片段，避免上下文过长导致评分漂移。

#### 4.1.2 完整阶段评分公式（数据驱动迭代）

基于 MVP 阶段积累的评分数据与用户反馈，通过回归分析确定各维度权重，引入更丰富的特征：

$$\text{FinalScore} = w_1 \times \text{KeyPersonBaseScore} + w_2 \times \text{LLMScore} \times 10 + w_3 \times \text{DurationScore} + w_4 \times \text{ContextScore} + w_5 \times \text{TimePeriodCoeff}$$

| 维度 | 说明 | 权重来源 |
|:---|:---|:---|
| KeyPersonBaseScore | 同 MVP 定义 | 回归确定 $w_1$ |
| LLMScore × 10 | 同 MVP 定义 | 回归确定 $w_2$ |
| DurationScore | 时长得分（对数饱和，见 4.2） | 回归确定 $w_3$ |
| ContextScore | 结构特征（轮次、发言比例等） | 回归确定 $w_4$ |
| TimePeriodCoeff | 时段系数（可配置，见 4.3） | 回归确定 $w_5$ |

**迭代策略**：MVP 上线后收集至少 2 周用户反馈数据（保留/删除/补充标注），以此为标签做逻辑回归，确定 $w_1$ 至 $w_5$ 的最优值。初始建议值：$w_1=1.0, w_2=1.0, w_3=0.3, w_4=0.2, w_5=0.1$。

### 4.2 DurationScore 设计（完整阶段）

采用**对数饱和曲线**替代原分段线性+衰减设计，避免长对话不合理降分：

$$\text{DurationScore} = 100 \times \frac{\ln(1 + \text{duration(min)})}{\ln(1 + 30)}$$

| 时长 | DurationScore | 说明 |
|:---|:---|:---|
| 1 分钟 | ≈ 20 | 短对话基础分 |
| 5 分钟 | ≈ 52 | 常规对话 |
| 15 分钟 | ≈ 81 | 中等长度会议 |
| 30 分钟 | **100** | 饱和参考点 |
| 60 分钟 | ≈ 113 | 超过 30 分钟仍缓慢增长，不衰减 |
| 120 分钟 | ≈ 127 | 长会议合理得分，不惩罚 |

设计原则：对话越长信息密度可能下降，但**不应惩罚**长时间的关键会议，因此采用对数饱和而非指数衰减。

### 4.3 时段系数（TimePeriodCoeff）

时段系数改为**可配置参数**，支持跨时区团队和非标准作息场景：

```yaml
# config/time_period_config.yaml
time_periods:
  - name: "早间会议"
    start: "08:00"
```

### 4.4 连续性保护

高分片段的前后相邻片段获得加成，形成"重要会话簇"，避免关键讨论的上下文被截断。

**连续性加分公式**：

$$\text{bonus} = \text{neighbor\_score} \times 0.3 \times \max\left(0,\ 1 - \frac{\text{interval}}{5}\right)$$

| 参数 | 说明 |
|:---|:---|
| neighbor_score | 相邻高分片段的 FinalScore |
| interval | 两片段之间的时间间隔（分钟） |
| 0.3 | 传播衰减系数，防止无限传播 |
| 5（分钟） | 最大有效间隔，超过 5 分钟不再传播 |

**计算规则**：
- 仅向低分方向传播（高分片段不从低分邻居获得加成）
- 单个片段最多从前、后各一个邻居获得 bonus，取较大值
- bonus 叠加后的总分不超过 180（MVP 阶段上限）

### 4.5 Top-K 与阈值筛选

#### 4.5.1 默认策略

| 参数 | 默认值 | 说明 |
|:---|:---|:---|
| 阈值（Threshold） | **60** | 低于此分数的片段不进入摘要 |
| Top-K | **20** | 超过阈值的片段中，最多保留 20 个最高分片段 |
| 实际生效规则 | **先过阈值，再取 Top-K** | 两道筛选串联执行 |

#### 4.5.2 与 Token 预算联动

$$K_{\text{effective}} = \min\left(K,\ \left\lfloor \frac{\text{token\_budget}}{\text{avg\_segment\_tokens}} \right\rfloor \right)$$

当 $K_{\text{effective}} < K$ 时，按 FinalScore 降序截断。日志中记录原始 K 与实际 $K_{\text{effective}}$，便于调参。

### 4.6 评分可解释性

每个片段保留完整的**评分分解记录**，用于调试、用户反馈分析和模型迭代：

### 4.7 LLM 隐式评估维度

LLM 在批量评分时综合考虑以下隐式特征，无需逐项拆分输出：

| 维度 | 关注点 | 高分信号示例 |
|:---|:---|:---|
| 语义重要性 | 决策指令、战略方向、资源分配 | "这个项目批了"、"预算调整为…" |
| 情感强度 | 紧急程度、争议程度、承诺程度 | "必须今天完成"、"我不同意这个方案" |
| 行动导向 | 待办事项、责任人、截止日期 | "你来负责这块，周五前交付" |
| 信息稀缺性 | 首次提及、独家信息、关键数据 | "刚收到消息，客户那边…" |

这些维度通过 Prompt 引导 LLM 综合判断，输出单一 1-10 分整数，不要求 LLM 逐维度打分，以确保调用轻量和输出稳定。

## 5. 多级摘要生成流程

### 5.0 架构总览：两级弹性摘要架构

Qwen3-Max 实际上下文窗口为 **262,144 tokens**。这一能力使得系统无需对所有场景都执行三级逐层压缩，而是可以根据当日数据规模动态选择最高效的生成路径。架构从"三级固定"演进为"两级弹性"，API 调用次数从 21+ 次降至 **1-4 次**，显著降低延迟与成本。

| 模式 | 触发条件 | 处理路径 | API 调用次数 | 覆盖场景占比 |
|:---|:---|:---|:---|:---|
| **短日模式** | 全天 tokens < 80K | 全量文本单次输入 → 直接生成全天日报 | **1 次** | ~90% |
| **长日模式** | 80K ≤ 全天 tokens ≤ 250K | 按时段分 2-3 片各生成时段摘要 → 合并生成全天日报 | **3-4 次** | ~10% |
| **降级模式** | 使用小模型（上下文 < 32K）或 Qwen3-Max 不可用 | 保留原三级架构：片段级 → 时段级 → 全天级 | **10+ 次** | 回退兜底 |

模式判定逻辑位于预处理层输出之后、摘要生成之前，由 `SummaryOrchestrator` 统一调度：

```python
def select_mode(total_tokens: int, model_context_limit: int) -> str:
    if model_context_limit < 64_000:
        return "degraded"      # 降级模式：三级架构
    if total_tokens < 80_000:
        return "short_day"     # 短日模式：单次直出
    if total_tokens <= 250_000:
        return "long_day"      # 长日模式：两级弹性
    return "degraded"          # 超长数据：回退三级
```

### 5.1 短日模式：单次调用直出全天日报

#### 5.1.1 适用场景与输入构造

短日模式覆盖约 **90% 的日常场景**——典型的 4-6 小时有效对话日，总文本量在 80K tokens 以内。系统将预处理后的全量"时间-说话人-内容"序列一次性送入 Qwen3-Max，由模型在单次推理中完成主题识别、信息抽取、去重整合与日报生成。

输入构造遵循以下拼接顺序：

| 区段 | 内容 | Token 预算 |
|:---|:---|:---|
| 系统提示词 | 角色定义 + 输出格式约束 + 反幻觉规则 | 1,500-2,000 |
| 关键人列表 | 用户消息最前端注入，含姓名、职务、关注领域 | 200-500 |
| 全量对话文本 | 时间戳 + 说话人 + 原始文本，按时间顺序排列 | ≤ 75,000 |
| 生成指令 | 明确输出结构与约束 | 200-300 |

#### 5.1.2 思考模式配置

短日模式启用 Qwen3-Max 的 **thinking 模式**：

```json
{
  "model": "qwen3-max",
  "enable_thinking": true,
  "thinking_budget": 16384,
  "messages": [
    {"role": "system", "content": "<系统提示词>"},
    {"role": "user", "content": "<关键人列表>\n<全量对话文本>\n<生成指令>"}
  ]
}
```

`thinking_budget` 设为 **16,384 tokens**，为模型提供充足的推理空间以完成主题聚类、跨时段关联、去重判断等复杂任务。

#### 5.1.3 关键事实直通通道

无论短日还是长日模式，系统均设置 **关键事实直通通道（Critical Fact Pass-through）**，确保以下三类高价值信息不经压缩、不被改写，直接保留至最终日报：

| 直通类型 | 识别规则 | 输出处理 |
|:---|:---|:---|
| **决策（Decision）** | 含"决定""确认""批准""通过"等动词 + 具体对象 | 原文引用，附时间戳与决策人 |
| **行动项（Action Item）** | 含责任人 + 任务描述 + 截止时间/优先级 | 结构化提取，保留原始表述 |
| **关键数据（Key Data）** | 数字、金额、百分比、日期等量化信息 | 原文引用，标注 ASR 置信度 |

提示词中通过显式指令实现直通：

> "对于明确的决策结论、分配的行动任务、以及涉及具体数字的关键数据，你必须保留原始表述，不得改写或概括。在输出中以 `[原文(置信度X.XX)：'...']` 格式标注。"

### 5.2 长日模式：两级弹性摘要

#### 5.2.1 适用场景与分片策略

长日模式处理 80K-250K tokens 的高密度工作日。系统按自然时段边界将全天对话切分为 **2-3 个时段片段（session）**，每片段控制在 80K tokens 以内。

分片策略优先级：

1. **自然间断优先**：利用 ≥30 分钟的静默间隔作为切分点
2. **时段均衡兜底**：若无明显间断，按上午/下午/晚间划分
3. **主题完整性约束**：切分点不得落在同一讨论主题的中间，需回退至最近的话题切换点

#### 5.2.2 第一级：时段摘要生成（Session Summarization）

每个时段片段独立送入 Qwen3-Max，启用思考模式生成时段摘要。`thinking_budget` 设为 **8,192 tokens**。

**输出规范**：

```json

  "time_range": "09:00-12:30",

  "topics": [
    {
      "topic_id": "topic_001",
      "title": "Q3预算分配方案",
      "duration_minutes": 45,
      "discussion_flow": "背景 → 方案对比 → 争议点 → 最终决策",
      "key_points": ["..."],
      "decisions": [
        {
          "content": "Q3研发预算确定为500万",
          "decision_maker": "张三",

        {

          "owner": "李四",

```

| 字段 | 说明 |
|:---|:---|
| `topic_id` | 讨论主题唯一标识，贯穿时段级到全天级，用于跨时段同一议题的识别与合并 |
| `decisions` | 独立提取的决策点，包含决策人、原文引用、时间戳 |
| `pending_topics` | 未得出结论的悬置议题，标注原因与建议的下一步行动 |
| `critical_facts` | 关键事实直通内容，不经压缩直接传递至全天级 |
| `source_refs` | 溯源链，标明本时段覆盖的原始片段范围 |
| `status` | 议题状态：`concluded`（已结论）/ `pending`（悬置）/ `ongoing`（进行中） |

#### 5.2.3 第二级：全天日报合并生成（Daily Report Synthesis）

将 2-3 个时段摘要的 JSON 输出作为输入，由 Qwen3-Max 合并生成全天日报。此步骤的核心任务：

1. **跨时段主题关联**：通过 `topic_id` 识别分布在不同时段的同一议题，合并讨论脉络
2. **去重与整合**：合并重复的行动项与决策，保留最完整版本
3. **全局排序**：按重要性而非时间顺序组织输出
4. **直通内容归集**：各时段的 `critical_facts` 汇入最终日报对应字段

`thinking_budget` 设为 **16,384 tokens**。

### 5.3 降级模式：三级架构兜底

当 Qwen3-Max 不可用需要使用小模型（上下文窗口 < 32K tokens）时，系统回退至三级架构：

| 级别 | 输入 | 输出 | 单次 Token 预算 |
|:---|:---|:---|:---|
| 第一级：片段摘要 | 单个对话片段（3-8K tokens） | 结构化片段摘要 JSON | ≤ 12K |
| 第二级：时段摘要 | 同一时段的片段摘要集合 | 时段摘要（同 5.2.2 格式） | ≤ 24K |
| 第三级：全天日报 | 各时段摘要集合 | 全天日报（同 5.4 格式） | ≤ 24K |

片段级摘要输出规范：

```json
{
  "chunk_id": "chunk_012",
  "summary": "200字内核心摘要",
  "key_points": ["要点1", "要点2", "要点3"],
  "action_items": [
    {"task": "任务描述", "owner": "责任人", "deadline": "截止日期", "priority": "high/medium/low"}
  ],
  "key_people_involved": ["张三", "李四"],
  "importance_score": 85,
  "importance_rationale": "P0级关键人参与，涉及战略决策",
  "source_refs": ["seg_001", "seg_002"]
}
```

降级模式保留 `topic_id` 贯穿机制和 `source_refs` 溯源链，确保输出格式与两级架构兼容。

### 5.4 全天日报输出规范

无论经由哪种模式生成，最终全天日报均遵循统一的分层输出结构，分为 **核心层（必生成）** 与 **扩展层（可选）**。

#### 5.4.1 核心层（必生成）

| 模块 | 内容 | 长度 |
|:---|:---|:---|
| `executive_summary` | 执行摘要，3-5 句话概述全天核心结论 | 50-100 字 |
| `key_meetings` | 关键会议/讨论回顾，按重要性排序；每条含 `decisions` 子字段 | 3-5 个，各 200-300 字 |
| `action_items` | 行动事项总表，按责任人组织，含任务、截止日期、优先级、来源引用 | 结构化表格 |

`key_meetings` 条目结构：

```json
{
  "title": "Q3预算评审会",
  "time_range": "10:00-11:30",
  "participants": ["张三", "李四", "王五"],
  "summary": "会议围绕Q3预算分配展开...",
  "decisions": [
    {
      "content": "Q3研发预算确定为500万",
      "decision_maker": "张三",
      "source_ref": "[原文(置信度0.95)：'Q3预算最终定500万']"
    }
  ],
  "topic_ids": ["topic_001", "topic_003"],
  "source_refs": ["seg_020", "seg_045"]
}
```

> 原 `decision_log` 合并入 `key_meetings` 作为 `decisions` 子字段，避免决策信息与会议上下文分离。

#### 5.4.2 扩展层（可选）

| 模块 | 内容 | 生成条件 |
|:---|:---|:---|
| `pending_topics` | 悬置议题列表：未得出结论的讨论，含原因与建议下一步 | 存在 status=pending 的议题时生成 |
| `follow_up` | 跟进建议：基于行动项与悬置议题推导的后续动作 | 存在行动项或悬置议题时生成 |
| `tomorrow_known_schedule` | 明日已知安排：**仅基于对话中明确提及的时间承诺**，不做推断 | 对话中存在明确时间承诺时生成 |

> `tomorrow_preview` 降级为 `tomorrow_known_schedule`，严格限定为对话中有明确证据的安排，杜绝推断性预测，降低幻觉风险。

#### 5.4.3 分层阅读体验

| 层级 | 内容 | 目标长度 | 适用场景 |
|:---|:---|:---|:---|
| **30 秒速览版** | `executive_summary` 精简版 | ~100 字 | 通知推送、快速浏览 |
| **3 分钟标准版** | executive_summary + key_meetings 标题与决策 + action_items 摘要 | 500-800 字 | 日常阅读、邮件摘要 |
| **完整存档版** | 全部核心层 + 扩展层，含完整溯源引用 | 1,500-3,000 字 | 归档、详细回顾、审计 |

### 5.5 topic_id 贯穿机制与去重策略

#### 5.5.1 topic_id 分配与跨时段关联

`topic_id` 是连接各级摘要的核心标识，确保同一讨论议题在不同处理阶段可被识别和合并：

| 阶段 | topic_id 行为 |
|:---|:---|
| 时段摘要生成 | 模型为每个独立讨论主题分配 `topic_id`（格式：`topic_XXX`） |
| 全天日报合并 | 通过 `topic_id` + 主题语义相似度识别跨时段的同一议题，合并讨论脉络 |
| 输出溯源 | 最终日报中每个 `key_meetings` 条目携带关联的 `topic_ids` 列表 |

短日模式下，模型在单次推理中内部完成 topic_id 的分配与关联，输出结构一致。

#### 5.5.2 去重策略

跨时段或跨片段的信息去重采用 **语义相似度 + 实体级校验** 双重机制：

| 维度 | 方法 | 阈值 |
|:---|:---|:---|
| 语义相似度 | 使用 **text2vec-base-chinese** 或 **m3e-large** 计算句向量余弦相似度 | **0.90-0.92** |
| 实体级校验 | 提取命名实体（人名、数字、日期），要求核心实体一致方可判定为重复 | 核心实体重合率 ≥ 80% |
| 行动项去重 | 责任人 + 任务关键词匹配，合并后保留最完整版本 | 责任人相同 + 任务相似度 ≥ 0.90 |

向量模型选型：**text2vec-base-chinese**（轻量~400MB）或 **m3e-large**（精度更高），均支持本地部署，延迟可控在 10ms 以内。

### 5.6 提示词工程规范

#### 5.6.1 系统角色定义

> "你是一位经验丰富的企业会议助理，曾为高管整理会议纪要。你深知哪些信息对决策者重要，擅长用简洁清晰的语言呈现关键结论和行动事项。你的工作客观准确，不添加未在原文中出现的信息。对于无法确认的内容，你会如实标注不确定性。"

系统提示词总预算放宽至 **1,500-2,000 tokens**。在 262K 上下文窗口下，系统提示词仅占不到 1%，无需严格压缩，应优先保障指令的完整性和清晰度。

#### 5.6.2 关键人强调机制

关键人列表注入位置为 **用户消息的最前端**，紧随系统提示词之后：

```
【关键人列表】
- 张三｜产品总监｜关注：产品规划、竞品分析
- 李四｜财务总监｜关注：预算、成本控制
- 王五｜研发负责人｜关注：技术方案、排期

请在摘要中优先关注以上人员的发言、决策与行动项。
---
<以下为全天对话记录>
```

#### 5.6.3 输出格式约束与校验规则

提示词中显式约束输出为 JSON 格式，并在末尾附加校验清单：

> "在输出前，请逐项检查：1）每个 action_item 是否包含 owner 和 task；2）每个 decision 是否包含 source_ref；3）executive_summary 是否在 100 字以内；4）所有关键数据是否标注了原文引用。"

#### 5.6.4 思考模式参数配置

| 场景 | thinking_budget | 说明 |
|:---|:---|:---|
| 短日模式（全天日报直出） | **16,384** | 需完成主题识别、跨时段关联、去重、排序等全部推理 |
| 长日模式 - 时段摘要 | **8,192** | 单时段推理复杂度较低 |
| 长日模式 - 全天合并 | **16,384** | 需完成跨时段关联与全局整合 |
| 降级模式 | 不启用思考模式 | 小模型通常不支持或效果不佳 |

### 5.7 反幻觉策略

#### 5.7.1 原文引用标记（含 ASR 置信度）

所有关键信息的引用标记需包含 ASR 置信度评分：

| 置信度区间 | 标注格式 | 处理方式 |
|:---|:---|:---|
| ≥ 0.90 | `[原文(置信度0.95)：'Q3预算500万']` | 正常引用，高可信 |
| 0.70 - 0.89 | `[原文(置信度0.78)：'大概是三百...万']` | 正常引用，中等可信 |
| < 0.70 | `[ASR识别不清晰，内容待确认：'...三...万...']` | 显式标注不确定性 |

#### 5.7.2 引用内容清洗规则

引用原文时执行 **最小化清洗**：

| 清洗操作 | 示例 |
|:---|:---|
| 去填充词 | "那个嗯就是说预算是500万" → "预算是500万" |
| 去口语重复 | "五百万五百万对五百万" → "五百万" |
| **不做**语义改写 | 不将"差不多五百万"改为"约500万" |
| **不做**数据补全 | 不将"五百"补全为"五百万"（除非上下文明确） |

#### 5.7.3 不确定性声明

- 推断内容标注：`[基于上下文推断]`
- 信息不完整标注：`[信息不完整，待核实]`
- 低置信度区域标注：`[ASR识别不清晰，内容待确认]`

### 5.8 溯源链设计

每级输出均包含 `source_refs` 字段，形成从最终日报到原始音频片段的完整溯源链：

```
全天日报 key_meetings[0].source_refs → ["seg_020", "seg_045"]
  ↓ 关联
时段摘要 topics[0].source_refs → segment_range ["seg_020", "seg_045"]
  ↓ 关联
原始片段 seg_020 → 音频时间戳 10:02:15 - 10:05:30
```

| 场景 | 使用方式 |
|:---|:---|
| 用户质疑某条决策 | 通过 `source_ref` 定位至原始对话片段，播放对应音频 |
| 审计合规要求 | 导出完整溯源链，证明摘要内容均有原文依据 |
| 摘要质量评估 | 抽样检查 `source_ref` 对应原文与摘要的一致性 |

### 6.2 ASR 噪声注入层

为缩小测试数据与真实 ASR 输出之间的差距，所有模拟文本在生成后须经过 ASR 噪声注入层处理。

#### 6.2.1 噪声类型定义

| 噪声类型 | 说明 | 示例 | 默认注入率 |
|:---|:---|:---|:---|
| 同音字替换 | 将词汇替换为发音相同/相近的错误词 | "预算" → "遇算" | 5%-10% 词级 |
| 断句错误 | 一句拆为两段，或两句合为一段 | "项目延期了。需要追加预算" → "项目延期了需要追加预算" | 8%-15% 句级 |
| 漏词/多词 | 丢失或重复关键词，尤其否定词和量词 | "不同意" → "同意" | 3%-8% 词级 |
| 填充词残留 | 保留 ASR 未清洗的口语填充词 | "嗯，那个，我觉得这个方案，呃，可以" | 10%-20% 句级 |
| 低置信度片段 | 大段不可识别内容 | "[inaudible_3.2s]...技术方案...确认..." | 2%-5% 片段级 |
| 说话人标签错误 | A/B 说话人标签互换 | Speaker A 的发言被标记为 Speaker B | 3%-8% 片段级 |

#### 6.2.2 噪声注入配置

```yaml
# config/noise_injection.yaml
noise_injection:
  enabled: true
  seed: 42
  global_error_rate: 0.10
  per_type:
    homophone_substitution:
      enabled: true
      rate: 0.08
    sentence_break_error:
      enabled: true
      rate: 0.10
    word_drop_insert:
      enabled: true
      rate: 0.05
      negation_bias: 0.3
    filler_word_residue:
      enabled: true
      rate: 0.15
    low_confidence_segment:
      enabled: true
      rate: 0.03
      min_duration_sec: 1.0
      max_duration_sec: 5.0
    speaker_label_error:
      enabled: true
      rate: 0.05
```

#### 6.2.3 噪声注入流程

1. **干净文本生成**：按 6.1 节场景与分布规则生成高质量模拟对话文本
2. **噪声叠加**：调用 `tests/noise_injector.py`，按配置对每条文本施加多层噪声
3. **标注保留**：原始干净文本作为 ground truth 保留，噪声版本标记各扰动位置与类型
4. **质量校验**：注入后抽样检查，确认错误率落在配置区间内

### 6.3 真实数据种子采集

为进一步缩小测试数据与生产数据的差距，需在系统上线前采集 **3-5 天真实 ASR 样本**作为种子数据。

| 步骤 | 内容 | 产出 |
|:---|:---|:---|
| 采集 | 从生产环境导出 3-5 天完整 ASR 转写结果 | 原始 ASR 语料库 |
| 脱敏 | 替换人名、公司名、敏感数字等 PII 信息 | 脱敏语料库 |
| 噪声分析 | 统计真实 ASR 中各类错误的实际分布比例 | 噪声分布报告 |
| 校准 | 根据真实分布反向校准噪声注入配置参数 | 更新后的 noise_injection.yaml |
| 入库 | 作为回归测试的固定数据集，纳入版本管理 | tests/seeds/ 目录 |

### 6.4 基准测试集（Benchmark）

#### 6.4.1 人工标注标准

采用**分层标注**策略，在保证标注质量的同时控制成本：

| 阶段 | 方式 | 说明 |
|:---|:---|:---|
| Pilot | 5 样本试标 | 3名标注员对同一批 5 个样本独立标注，对齐标准，计算初始 Kappa 值 |
| 全量标注 | 1 人全标 | 每个样本由 1 名标注员完成全量标注 |
| 分歧复核 | 2-3 人复核 | 评分极端或歧义片段增加复核 |
| 仲裁 | 专家裁定 | 复核仍不一致的片段由业务专家最终裁定 |

标注维度：
- **重要性标签**：每片段 1-5 分
- **摘要质量评分**：完整性、准确性、简洁性、可操作性四维度
- **关键信息清单**：预定义关键信息点列表，逐项标记是否被摘要覆盖
- **行动项标注**：标记原文中所有行动项（含责任人、时间节点）

#### 6.4.2 评估指标体系

**一级指标（核心质量指标）**：

| 指标 | 优先级 | 定义 | 目标值 |
|:---|:---|:---|:---|
| 关键信息召回率 | **P0** | 预定义关键信息点被摘要覆盖的比例 | ≥ 95% |
| 行动项 F1 | **P0** | 行动项准确率与召回率的调和平均 | F1 ≥ 92% |
| 关键人归属准确率 | P1 | 关键人发言、决策在摘要中归属正确的比例 | ≥ 93% |
| 事实一致性 | P1 | 摘要中事实与原文一致的比例 | ≥ 95% |
| 遗漏严重度加权召回 | P1 | 按业务严重度加权的召回率（P0=3, P1=2, P2=1） | ≥ 90% |

**二级指标（辅助参考指标）**：

| 指标 | 说明 | 目标值 |
|:---|:---|:---|
| 关键人召回率 | P0级对话 100% 保留，P1级 >95% | 100% / >95% |
| 重要片段准确率 | 保留片段中真正重要的比例 | >90% |
| 信息压缩比 | 原文长度与摘要长度之比 | 30-50:1 |
| ROUGE-L | 最长公共子序列 F1 | 参考值 |
| BERTScore | 语义相似度 F1 | 参考值 |
| 人工满意度 | 业务专家整体评分 | ≥ 4.0 |

#### 6.4.3 对比基线

| 方法 | 说明 | 预期表现 |
|:---|:---|:---|
| 纯时长过滤 | 保留 >10 分钟片段 | 低召回，高精确 |
| 纯关键人过滤 | 仅保留关键人对话 | 高召回关键人，遗漏重要非关键人对话 |
| 随机采样 | 随机保留 20% 片段 | 基准下限 |
| **本方案** | 多维度融合评分 | **最优综合表现** |

### 7.3 错误处理与降级

| 异常场景 | 处理策略 | 降级路径 |
|:---|:---|:---|
| 模型调用失败 | 指数退避重试（1s, 2s, 4s, 8s） | 3 次失败后降级至 qwen-plus |
| Batch API 超时 | 超过 SLA 时间未返回结果 | 自动切换为实时级逐条调用 |
| Token 超限 | 动态分片压缩，关键内容优先 | 压缩至最小单元仍超限则人工队列 |
| 解析失败 | 正则兜底提取关键字段 | 原始内容标记输出，人工复核 |
| 配置加载失败 | 启动时校验配置文件格式 | 使用上一次有效配置，告警通知 |
