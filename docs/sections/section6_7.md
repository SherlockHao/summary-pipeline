### 6.2 ASR 噪声注入层

为缩小测试数据与真实 ASR 输出之间的差距，所有模拟文本在生成后须经过 ASR 噪声注入层处理。噪声类型和注入比例均通过配置文件控制，支持按场景独立调节。

#### 6.2.1 噪声类型定义

| 噪声类型 | 说明 | 示例 | 默认注入率 |
|:---|:---|:---|:---|
| 同音字替换 | 将词汇替换为发音相同/相近的错误词 | "预算" → "遇算"；"通过" → "同过" | 5%-10% 词级 |
| 断句错误 | 一句拆为两段，或两句合为一段 | "项目延期了。需要追加预算" → "项目延期了需要追加预算" | 8%-15% 句级 |
| 漏词/多词 | 丢失或重复关键词，尤其否定词和量词 | "不同意" → "同意"；"三百万" → "三万" | 3%-8% 词级 |
| 填充词残留 | 保留 ASR 未清洗的口语填充词 | "嗯，那个，我觉得这个方案，呃，可以" | 10%-20% 句级 |
| 低置信度片段 | 大段不可识别内容，模拟信号丢失或极强噪声 | "[inaudible_3.2s]...技术方案...确认...[inaudible]" | 2%-5% 片段级 |
| 说话人标签错误 | A/B 说话人标签互换或归属错误 | Speaker A 的发言被标记为 Speaker B | 3%-8% 片段级 |

#### 6.2.2 噪声注入配置

```yaml
# config/noise_injection.yaml
noise_injection:
  enabled: true
  seed: 42                        # 随机种子，保证可复现
  global_error_rate: 0.10         # 全局基准错误率
  per_type:
    homophone_substitution:
      enabled: true
      rate: 0.08
    sentence_break_error:
      enabled: true
      rate: 0.10
    word_drop_insert:
      enabled: true
      rate: 0.05
      negation_bias: 0.3          # 否定词被漏掉的额外概率
    filler_word_residue:
      enabled: true
      rate: 0.15
    low_confidence_segment:
      enabled: true
      rate: 0.03
      min_duration_sec: 1.0
      max_duration_sec: 5.0
    speaker_label_error:
      enabled: true
      rate: 0.05
```

#### 6.2.3 噪声注入流程

1. **干净文本生成**：按 6.1 节场景与分布规则生成高质量模拟对话文本。
2. **噪声叠加**：调用 `tests/noise_injector.py`，按配置对每条文本施加多层噪声。各噪声类型独立采样，可叠加。
3. **标注保留**：原始干净文本作为 ground truth 保留，噪声版本标记各扰动位置与类型，供后续评估溯源。
4. **质量校验**：注入后抽样检查，确认错误率落在配置区间内。

### 6.3 真实数据种子采集

为进一步缩小测试数据与生产数据的差距，需在系统上线前采集 **3-5 天真实 ASR 样本**作为种子数据。

| 步骤 | 内容 | 产出 |
|:---|:---|:---|
| 采集 | 从生产环境导出 3-5 天完整 ASR 转写结果（含时间戳、说话人标签） | 原始 ASR 语料库 |

### 6.4 基准测试集（Benchmark）

#### 6.4.1 人工标注标准

采用**分层标注**策略，在保证标注质量的同时控制成本：

| 阶段 | 方式 | 说明 |
|:---|:---|:---|
| Pilot | 5 样本试标 | 全体标注员（3人）对同一批 5 个样本独立标注，对齐标注标准，计算初始 Kappa 值 |
| 全量标注 | 1 人全标 | 每个样本由 1 名标注员完成全量标注（重要性标签、关键信息清单、摘要质量评分） |
| 分歧复核 | 2-3 人复核 | 对标注中存在歧义或评分极端（重要性评分 2-4 分区间）的片段，增加 2-3 名标注员复核，取多数一致结果 |
| 仲裁 | 专家裁定 | 复核仍不一致的片段由业务专家最终裁定 |

标注维度：
- **重要性标签**：每片段 1-5 分
- **摘要质量评分**：完整性、准确性、简洁性、可操作性四维度
- **关键信息清单**：预定义关键信息点列表，逐项标记是否被摘要覆盖
- **行动项标注**：标记原文中所有行动项（含责任人、时间节点），对应检查摘要输出

#### 6.4.2 评估指标体系

**一级指标（核心质量指标）**：

| 指标 | 优先级 | 定义 | 目标值 | 计算方式 |
|:---|:---|:---|:---|:---|
| 关键信息召回率 | **P0** | 预定义关键信息点被摘要覆盖的比例 | >= 95% | 召回信息点数 / 总关键信息点数 |
| 行动项 F1 | **P0** | 行动项的准确率与召回率的调和平均 | Precision >= 90%, Recall >= 95%, F1 >= 92% | 2PR/(P+R)，行动项按（内容+责任人+时限）三元组匹配 |
| 关键人归属准确率 | P1 | 关键人发言、决策、指示在摘要中归属正确的比例 | >= 93% | 归属正确条目数 / 总关键人相关条目数 |
| 事实一致性 | P1 | 摘要中陈述的事实与原文一致的比例 | >= 95% | 人工逐条核实 + NLI 模型辅助 |
| 遗漏严重度加权召回 | P1 | 按遗漏信息的业务严重度加权的召回率 | >= 90% | sum(weight_i * recalled_i) / sum(weight_i)，权重按 P0=3, P1=2, P2=1 |

**二级指标（辅助参考指标）**：

| 指标 | 说明 | 目标值 | 计算方式 |
|:---|:---|:---|:---|
| 关键人召回率 | P0 级对话 100% 保留，P1 级 >95% | 100% / >95% | 保留的关键人片段数 / 总关键人片段数 |
| 重要片段准确率 | 保留片段中真正重要的比例 | >90% | 真正重要片段数 / 保留片段总数 |
| 信息压缩比 | 原文长度与摘要长度之比 | 30-50:1 | 原文 token 数 / 摘要 token 数 |
| ROUGE-L | 摘要与参考摘要的最长公共子序列 | 参考值 | F1 分数 |
| BERTScore | 基于语义相似度的生成质量 | 参考值 | F1 分数 |
| 人工满意度 | 业务专家整体评分 | >= 4.0 | 1-5 分 Likert 量表 |

#### 6.4.3 对比基线

| 方法 | 说明 | 预期表现 |
|:---|:---|:---|
| 纯时长过滤 | 保留 >10 分钟片段 | 低召回，高精确 |
| 纯关键人过滤 | 仅保留关键人对话 | 高召回关键人，遗漏重要非关键人对话 |
| 随机采样 | 随机保留 20% 片段 | 基准下限 |
| **本方案** | 多维度融合评分 | **最优综合表现** |

### 6.5 端到端 Golden Case

由业务专家编写 5-10 个端到端 golden case，作为回归测试的核心基准。每个 case 包含完整的输入-输出对照。

| 编号 | 场景描述 | 输入 | 期望输出 | 覆盖要点 |
|:---|:---|:---|:---|:---|
| GC-01 | CEO 一对一沟通 + 决策 | 含 P0 关键人的 15 分钟对话（含 ASR 噪声） | 完整日报片段：决策内容、行动项、时间节点 | 关键人识别、行动项提取、噪声鲁棒性 |
| GC-02 | 跨部门会议 + 多议题 | 6 人参与的 45 分钟会议，3 个议题 | 按议题分段摘要，各议题结论与后续行动 | 长对话分片、多议题分割、信息压缩 |
| GC-03 | 全天综合日报 | 一天 8 段对话的完整输入 | 结构化日报（关键决策、行动项、信息同步） | 全链路端到端、去重、优先级排序 |
| GC-04 | ASR 高噪声场景 | 注入 15%-20% 错误率的对话 | 正确还原核心信息，标记不确定内容 | 噪声鲁棒性、否定词丢失检测 |
| GC-05 | 关键人全天未出现 | 仅含 P2-P3 级人员的对话集 | 正常生成日报，无关键人相关内容标注为"未参与" | 边界处理、空结果优雅降级 |
| GC-06 | 中英文混合会议 | 技术讨论中大量英文术语和代码片段 | 术语保留原文，摘要以中文为主 | 多语言处理 |
| GC-07 | 说话人标签错乱 | 同一段对话中 20% 标签互换 | 基于上下文修正归属，标记低置信度归属 | 说话人纠错能力 |
| GC-08 | 极短碎片聚合 | 20 段 <2 分钟的碎片对话 | 合理聚合为 3-5 个主题组，各组独立摘要 | 碎片聚合、会话检测 |

Golden case 管理规范：
- 存储位置：`tests/e2e/golden_cases/`
- 格式：每个 case 为独立 YAML 文件，包含 `input`、`expected_output`、`metadata` 三个顶层字段
- 维护：每次模型或 prompt 升级后必须全量回归，结果差异需人工确认

|:---|:---|:---|

| 跨天录音（跨午夜） | 会议从 23:30 持续到 00:30，跨越日期边界 | 按会议完整性归属到会议开始日期；若需拆分则在两天日报中均体现，注明跨天 |

#### 7.1.1 新增模块说明

#### 7.1.2 两级弹性架构说明

系统采用两级调用策略以平衡实时性与成本：

| 级别 | 调用方式 | 适用场景 | 入口模块 |
|:---|:---|:---|:---|
| 实时级 | `qwen_client.py` 单条调用 | 日间增量摘要、紧急对话处理 | `api/qwen_client.py` |
| 批量级 | `batch_client.py` Batch API | 日终全量日报生成、历史数据回刷 | `api/batch_client.py` |

降级路径：Batch API 不可用时，自动回退到实时级逐条调用，由 `retry_handler.py` 统一管控。`streaming.py` 作为可选模块，仅在需要流式输出的前端场景中启用，非核心数据流路径。

### 7.2 配置管理

#### 7.2.1 关键人配置（key_people.yaml）

关键人配置文件采用 **Git 版本管理**，不依赖外部配置中心：

```yaml
# config/key_people.yaml
version: "2026-03-27"
people:
  - name: "张三"
    aliases: ["张总", "三哥"]
    level: P0
    department: "CEO办公室"
    active: true
  - name: "李四"
    aliases: ["李总监"]
    level: P1
    department: "技术部"
    active: true
  - name: "王五"
    aliases: ["王经理"]
    level: P1
    department: "产品部"
    active: false           # 已离职，保留历史配置
    inactive_since: "2026-03-15"
```

加载策略：
- **启动时加载**：应用启动时读取 `key_people.yaml` 并缓存到内存
- **手动 reload**：通过管理接口或命令行触发配置重新加载，无需重启服务
- **变更追踪**：所有配置变更通过 Git commit 记录，支持审计与回滚

```python
# 手动 reload 示例
class ConfigManager:
    def reload_key_people(self):
        """手动触发关键人配置重新加载"""
        self._key_people = self._load_yaml("config/key_people.yaml")
        logger.info(f"关键人配置已重载，版本: {self._key_people['version']}, "
                    f"有效人数: {sum(1 for p in self._key_people['people'] if p['active'])}")
```

### 7.3 错误处理与降级

| 异常场景 | 处理策略 | 降级路径 |
|:---|:---|:---|
| 模型调用失败 | 指数退避重试（1s, 2s, 4s, 8s） | 3 次失败后降级至 qwen-plus |
| Batch API 超时 | 超过 SLA 时间未返回结果 | 自动切换为实时级逐条调用 |
| Token 超限 | 动态分片压缩，关键内容优先 | 压缩至最小单元仍超限则人工队列 |
| 解析失败 | 正则兜底提取关键字段 | 原始内容标记输出，人工复核 |
| 配置加载失败 | 启动时校验配置文件格式 | 使用上一次有效配置，告警通知 |

### 7.4 性能优化

| 优化点 | 策略 | 效果 |
|:---|:---|:---|
| 并发控制 | 异步批量调用，适配 600 RPM 速率限制 | 吞吐量最大化 |
| Batch API 降本 | 非实时任务走 Batch API，利用低优先级队列 | 降低 30%-50% API 成本 |
| 缓存策略 | 重复片段 MD5 哈希去重，摘要结果 LRU 缓存 | 减少 30%-50% 冗余调用 |
| 成本监控 | 实时 Token 消耗追踪，预算告警 | 成本可控 |
| Prompt 模板化 | 提示词从代码分离，支持热更新与 A/B 测试 | 迭代效率提升 |
