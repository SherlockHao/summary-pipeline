## 5. 多级摘要生成流程

### 5.0 架构总览：两级弹性摘要架构

Qwen3-Max 实际上下文窗口为 **262,144 tokens**，远超此前预估的 20,000 tokens。这一能力使得系统无需对所有场景都执行三级逐层压缩，而是可以根据当日数据规模动态选择最高效的生成路径。架构从"三级固定"演进为"两级弹性"，API 调用次数从 21+ 次降至 **1-4 次**，显著降低延迟与成本。

| 模式 | 触发条件 | 处理路径 | API 调用次数 | 覆盖场景占比 |
|:---|:---|:---|:---|:---|
| **短日模式** | 全天 tokens < 80K | 全量文本单次输入 → 直接生成全天日报 | **1 次** | ~90% |
| **长日模式** | 80K ≤ 全天 tokens ≤ 250K | 按时段分 2-3 片各生成时段摘要 → 合并生成全天日报 | **3-4 次** | ~10% |
| **降级模式** | 使用小模型（上下文 < 32K）或 Qwen3-Max 不可用 | 保留原三级架构：片段级 → 时段级 → 全天级 | **10+ 次** | 回退兜底 |

模式判定逻辑位于预处理层输出之后、摘要生成之前，由 `SummaryOrchestrator` 统一调度：

```python
def select_mode(total_tokens: int, model_context_limit: int) -> str:
    if model_context_limit < 64_000:
        return "degraded"      # 降级模式：三级架构
    if total_tokens < 80_000:
        return "short_day"     # 短日模式：单次直出
    if total_tokens <= 250_000:
        return "long_day"      # 长日模式：两级弹性
    return "degraded"          # 超长数据：回退三级
```

### 5.1 短日模式：单次调用直出全天日报

#### 5.1.1 适用场景与输入构造

短日模式覆盖约 **90% 的日常场景**——典型的 4-6 小时有效对话日，总文本量在 80K tokens 以内。系统将预处理后的全量"时间-说话人-内容"序列一次性送入 Qwen3-Max，由模型在单次推理中完成主题识别、信息抽取、去重整合与日报生成。

输入构造遵循以下拼接顺序（总预算控制在 80K tokens 以内）：

| 区段 | 内容 | Token 预算 |
|:---|:---|:---|
| 系统提示词 | 角色定义 + 输出格式约束 + 反幻觉规则 | 1,500-2,000 |
| 关键人列表 | 用户消息最前端注入，含姓名、职务、关注领域 | 200-500 |
| 全量对话文本 | 时间戳 + 说话人 + 原始文本，按时间顺序排列 | ≤ 75,000 |
| 生成指令 | 明确输出结构与约束 | 200-300 |

#### 5.1.2 思考模式配置

短日模式启用 Qwen3-Max 的 **thinking 模式**，允许模型在生成最终结构化输出之前进行充分的内部推理：

```json
{
  "model": "qwen3-max",
  "enable_thinking": true,
  "thinking_budget": 16384,
  "messages": [
    {"role": "system", "content": "<系统提示词>"},
    {"role": "user", "content": "<关键人列表>\n<全量对话文本>\n<生成指令>"}
  ]
}
```

`thinking_budget` 设为 **16,384 tokens**，为模型提供充足的推理空间以完成主题聚类、跨时段关联、去重判断等复杂任务。

#### 5.1.3 关键事实直通通道

无论短日还是长日模式，系统均设置 **关键事实直通通道（Critical Fact Pass-through）**，确保以下三类高价值信息不经压缩、不被改写，直接保留至最终日报：

| 直通类型 | 识别规则 | 输出处理 |
|:---|:---|:---|
| **决策（Decision）** | 含"决定""确认""批准""通过"等动词 + 具体对象 | 原文引用，附时间戳与决策人 |
| **行动项（Action Item）** | 含责任人 + 任务描述 + 截止时间/优先级 | 结构化提取，保留原始表述 |
| **关键数据（Key Data）** | 数字、金额、百分比、日期等量化信息 | 原文引用，标注 ASR 置信度 |

提示词中通过显式指令实现直通：

> "对于明确的决策结论、分配的行动任务、以及涉及具体数字的关键数据，你必须保留原始表述，不得改写或概括。在输出中以 `[原文(置信度X.XX)：'...']` 格式标注。"

---

### 5.2 长日模式：两级弹性摘要

#### 5.2.1 适用场景与分片策略

长日模式处理 80K-250K tokens 的高密度工作日。系统按自然时段边界将全天对话切分为 **2-3 个时段片段（session）**，每片段控制在 80K tokens 以内，确保单次调用不超出高效处理范围。

分片策略优先级：

1. **自然间断优先**：利用 ≥30 分钟的静默间隔作为切分点
2. **时段均衡兜底**：若无明显间断，按上午/下午/晚间划分
3. **主题完整性约束**：切分点不得落在同一讨论主题的中间，需回退至最近的话题切换点

#### 5.2.2 第一级：时段摘要生成（Session Summarization）

每个时段片段独立送入 Qwen3-Max，启用思考模式生成时段摘要。

**思考模式配置**：`thinking_budget` 设为 **8,192 tokens**（时段级推理复杂度低于全天级）。

**输入构造**与短日模式类似，区别在于对话文本仅包含当前时段的片段，且生成指令要求输出时段级中间结构而非最终日报。

**输出规范**：

```json
{
  "session_id": "session_morning",
  "time_range": "09:00-12:30",
  "session_theme": "产品评审与Q3预算",
  "participants": ["张三(产品总监)", "李四(财务)", "王五(研发)"],
  "topics": [
    {
      "topic_id": "topic_001",
      "title": "Q3预算分配方案",
      "duration_minutes": 45,
      "discussion_flow": "背景 → 方案对比 → 争议点 → 最终决策",
      "key_points": ["...", "..."],
      "decisions": [
        {
          "content": "Q3研发预算确定为500万",
          "decision_maker": "张三",
          "source_ref": "[原文(置信度0.95)：'Q3预算最终定500万']",
          "timestamp": "10:32"
        }
      ],
      "action_items": [
        {
          "task": "输出Q3预算明细表",
          "owner": "李四",
          "deadline": "本周五",
          "priority": "high",
          "source_ref": "seg_042"
        }
      ],
      "status": "concluded"
    },
    {
      "topic_id": "topic_002",
      "title": "新功能优先级排序",
      "duration_minutes": 30,
      "discussion_flow": "需求列举 → 初步排序 → 未达成一致",
      "key_points": ["..."],
      "decisions": [],
      "action_items": [],
      "status": "pending"
    }
  ],
  "pending_topics": [
    {
      "topic_id": "topic_002",
      "reason": "参会者对优先级未达成一致，待下次会议继续讨论",
      "next_step": "张三整理各方意见后重新组织讨论"
    }
  ],
  "critical_facts": [
    {
      "type": "key_data",
      "content": "Q3预算500万",
      "source_ref": "[原文(置信度0.95)：'Q3预算最终定500万']"
    }
  ],
  "source_refs": {
    "segment_range": ["seg_001", "seg_058"],
    "total_segments": 58
  }
}
```

**字段说明**：

| 字段 | 说明 |
|:---|:---|
| `topic_id` | 讨论主题唯一标识，贯穿时段级到全天级，用于跨时段同一议题的识别与合并 |
| `decisions` | 从 `key_discussions` 中独立提取的决策点，包含决策人、原文引用、时间戳 |
| `pending_topics` | 未得出结论的悬置议题，标注原因与建议的下一步行动 |
| `critical_facts` | 关键事实直通内容，不经压缩直接传递至全天级 |
| `source_refs` | 溯源链，标明本时段覆盖的原始片段范围，支持逐级回溯 |
| `status` | 议题状态：`concluded`（已结论）/ `pending`（悬置）/ `ongoing`（进行中） |

#### 5.2.3 第二级：全天日报合并生成（Daily Report Synthesis）

将 2-3 个时段摘要的 JSON 输出作为输入，由 Qwen3-Max 合并生成全天日报。此步骤的核心任务：

1. **跨时段主题关联**：通过 `topic_id` 识别分布在不同时段的同一议题，合并讨论脉络
2. **去重与整合**：合并重复的行动项与决策，保留最完整版本
3. **全局排序**：按重要性而非时间顺序组织输出
4. **直通内容归集**：各时段的 `critical_facts` 汇入最终日报对应字段

**思考模式配置**：`thinking_budget` 设为 **16,384 tokens**。

### 5.3 降级模式：三级架构兜底

当 Qwen3-Max 不可用需要使用小模型（上下文窗口 < 32K tokens）时，系统回退至三级架构：

| 级别 | 输入 | 输出 | 单次 Token 预算 |
|:---|:---|:---|:---|
| 第一级：片段摘要 | 单个对话片段（3-8K tokens） | 结构化片段摘要 JSON | ≤ 12K |
| 第二级：时段摘要 | 同一时段的片段摘要集合 | 时段摘要（同 5.2.2 格式） | ≤ 24K |
| 第三级：全天日报 | 各时段摘要集合 | 全天日报（同 5.4 格式） | ≤ 24K |

片段级摘要输出规范：

```json
{
  "chunk_id": "chunk_012",
  "summary": "200字内核心摘要",
  "key_points": ["要点1", "要点2", "要点3"],
  "action_items": [
    {
      "task": "任务描述",
      "owner": "责任人",
      "deadline": "截止日期",
      "priority": "high/medium/low"
    }
  ],
  "key_people_involved": ["张三", "李四"],
  "importance_score": 85,
  "importance_rationale": "P0级关键人参与，涉及战略决策",
  "source_refs": ["seg_001", "seg_002"]
}
```

降级模式保留 `topic_id` 贯穿机制和 `source_refs` 溯源链，确保输出格式与两级架构兼容。

### 5.4 全天日报输出规范

无论经由哪种模式生成，最终全天日报均遵循统一的分层输出结构，分为 **核心层（必生成）** 与 **扩展层（可选）**。

#### 5.4.1 核心层（必生成）

| 模块 | 内容 | 长度 |
|:---|:---|:---|
| `executive_summary` | 执行摘要，3-5 句话概述全天核心结论 | 50-100 字 |
| `key_meetings` | 关键会议/讨论回顾，按重要性排序；每个条目包含 `decisions` 子字段记录该会议的决策点 | 3-5 个，各 200-300 字 |
| `action_items` | 行动事项总表，按责任人组织，含任务、截止日期、优先级、来源引用 | 结构化表格 |

`key_meetings` 条目结构：

```json
{
  "title": "Q3预算评审会",
  "time_range": "10:00-11:30",
  "participants": ["张三", "李四", "王五"],
  "summary": "会议围绕Q3预算分配展开...",
  "decisions": [
    {
      "content": "Q3研发预算确定为500万",
      "decision_maker": "张三",
      "source_ref": "[原文(置信度0.95)：'Q3预算最终定500万']"
    }
  ],
  "topic_ids": ["topic_001", "topic_003"],
  "source_refs": ["seg_020", "seg_045"]
}
```

> **设计说明**：原 `decision_log` 合并入 `key_meetings` 作为 `decisions` 子字段，避免决策信息与会议上下文分离，提升阅读连贯性。

#### 5.4.2 扩展层（可选）

| 模块 | 内容 | 生成条件 |
|:---|:---|:---|
| `pending_topics` | 悬置议题列表：未得出结论的讨论，含原因与建议下一步 | 存在 status=pending 的议题时生成 |
| `follow_up` | 跟进建议：基于行动项与悬置议题推导的后续动作 | 存在行动项或悬置议题时生成 |
| `tomorrow_known_schedule` | 明日已知安排：**仅基于对话中明确提及的时间承诺**（如"明天下午两点开会"），不做推断 | 对话中存在明确时间承诺时生成 |

> **设计说明**：`tomorrow_preview` 降级为 `tomorrow_known_schedule`，严格限定为对话中有明确证据的安排，杜绝基于待办事项的推断性预测，降低幻觉风险。

#### 5.4.3 分层阅读体验

系统为同一份日报提供三种阅读粒度，适配不同场景：

| 层级 | 内容 | 目标长度 | 适用场景 |
|:---|:---|:---|:---|
| **30 秒速览版** | `executive_summary` 精简版，一段话概括全天 | ~100 字 | 通知推送、快速浏览 |
| **3 分钟标准版** | `executive_summary` + `key_meetings` 标题与决策 + `action_items` 摘要 | 500-800 字 | 日常阅读、邮件摘要 |
| **完整存档版** | 全部核心层 + 扩展层，含完整溯源引用 | 1,500-3,000 字 | 归档、详细回顾、审计 |

30 秒速览版通过对 `executive_summary` 进一步压缩生成，不额外调用 API，由前端模板或简单规则裁剪实现。

### 5.5 topic_id 贯穿机制与去重策略

#### 5.5.1 topic_id 分配与跨时段关联

`topic_id` 是连接各级摘要的核心标识，确保同一讨论议题在不同处理阶段可被识别和合并：

| 阶段 | topic_id 行为 |
|:---|:---|
| 时段摘要生成 | 模型为每个独立讨论主题分配 `topic_id`（格式：`topic_XXX`） |
| 全天日报合并 | 模型通过 `topic_id` + 主题语义相似度识别跨时段的同一议题，合并讨论脉络 |
| 输出溯源 | 最终日报中每个 `key_meetings` 条目携带关联的 `topic_ids` 列表，支持回溯至时段级原文 |

短日模式下，模型在单次推理中内部完成 topic_id 的分配与关联，输出结构一致。

#### 5.5.2 去重策略

跨时段或跨片段的信息去重采用 **语义相似度 + 实体级校验** 双重机制：

| 维度 | 方法 | 阈值 |
|:---|:---|:---|
| 语义相似度 | 使用 **text2vec-base-chinese** 或 **m3e-large** 计算句向量余弦相似度 | **0.90-0.92**（高于通用阈值 0.85，减少误去重） |
| 实体级校验 | 提取两条信息中的命名实体（人名、数字、日期），要求核心实体一致方可判定为重复 | 核心实体重合率 ≥ 80% |
| 行动项去重 | 责任人 + 任务关键词匹配，合并后保留最完整版本（含最明确的截止日期与优先级） | 责任人相同 + 任务相似度 ≥ 0.90 |

向量模型选型说明：

- **text2vec-base-chinese**：轻量（~400MB），推理快，适合部署资源有限的场景
- **m3e-large**：精度更高，适合对去重质量要求严格的场景
- 两者均支持本地部署，无需外部 API 调用，延迟可控在 10ms 以内

### 5.6 提示词工程规范

#### 5.6.1 系统角色定义

> "你是一位经验丰富的企业会议助理，曾为高管整理会议纪要。你深知哪些信息对决策者重要，擅长用简洁清晰的语言呈现关键结论和行动事项。你的工作客观准确，不添加未在原文中出现的信息。对于无法确认的内容，你会如实标注不确定性。"

系统提示词总预算放宽至 **1,500-2,000 tokens**。在 262K 上下文窗口下，系统提示词仅占不到 1%，无需严格压缩，应优先保障指令的完整性和清晰度。

#### 5.6.2 关键人强调机制

关键人列表注入位置为 **用户消息的最前端**，紧随系统提示词之后，确保模型在处理对话文本前已建立关键人认知：

```
【关键人列表】
- 张三｜产品总监｜关注：产品规划、竞品分析
- 李四｜财务总监｜关注：预算、成本控制
- 王五｜研发负责人｜关注：技术方案、排期

请在摘要中优先关注以上人员的发言、决策与行动项。
---
<以下为全天对话记录>
```

#### 5.6.3 输出格式约束与校验规则

提示词中显式约束输出为 JSON 格式，并在系统提示词末尾附加校验清单：

> "在输出前，请逐项检查：1）每个 action_item 是否包含 owner 和 task；2）每个 decision 是否包含 source_ref；3）executive_summary 是否在 100 字以内；4）所有关键数据是否标注了原文引用。"

#### 5.6.4 思考模式参数配置

| 场景 | thinking_budget | 说明 |
|:---|:---|:---|
| 短日模式（全天日报直出） | **16,384** | 需完成主题识别、跨时段关联、去重、排序等全部推理 |
| 长日模式 - 时段摘要 | **8,192** | 单时段推理复杂度较低 |
| 长日模式 - 全天合并 | **16,384** | 需完成跨时段关联与全局整合 |
| 降级模式 | 不启用思考模式 | 小模型通常不支持或效果不佳 |

### 5.7 反幻觉策略

#### 5.7.1 原文引用标记（含 ASR 置信度）

所有关键信息的引用标记需包含 ASR 置信度评分，帮助读者判断信息可靠性：

| 置信度区间 | 标注格式 | 处理方式 |
|:---|:---|:---|
| ≥ 0.90 | `[原文(置信度0.95)：'Q3预算500万']` | 正常引用，高可信 |
| 0.70 - 0.89 | `[原文(置信度0.78)：'大概是三百...万']` | 正常引用，中等可信 |
| < 0.70 | `[ASR识别不清晰，内容待确认：'...三...万...']` | 显式标注不确定性，提醒读者核实 |

#### 5.7.2 引用内容清洗规则

引用原文时执行 **最小化清洗**，在保持原意的前提下提升可读性：

| 清洗操作 | 示例 |
|:---|:---|
| 去填充词 | "那个嗯就是说预算是500万" → "预算是500万" |
| 去口语重复 | "五百万五百万对五百万" → "五百万" |
| **不做**语义改写 | 不将"差不多五百万"改为"约500万" |
| **不做**数据补全 | 不将"五百"补全为"五百万"（除非上下文明确） |

#### 5.7.3 不确定性声明

- 推断内容标注：`[基于上下文推断]`
- 信息不完整标注：`[信息不完整，待核实]`
- 低置信度区域标注：`[ASR识别不清晰，内容待确认]`

---

### 5.8 溯源链设计

每级输出均包含 `source_refs` 字段，形成从最终日报到原始音频片段的完整溯源链：

